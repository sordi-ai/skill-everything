Skill-Everythink

"Agents that never make the same mistake twice"

Git-Versioned Agent Memory — Der intelligente Feintuning-Ersatz

Open Source • MIT Lizenz • Agent-agnostisch

Das Problem

AI-Agenten vergessen alles. Jede Session startet bei Null. Existierende Lösungen haben fundamentale Schwächen:

  • 💸 Fine-Tuning — Teuer ($$$), starr, veraltet schnell, Vendor Lock-in
  • 🏗️ RAG — Braucht Infrastruktur: Vektordatenbanken, Embedding-APIs, Maintenance
  • 📄 Custom Instructions — Statisch, nicht versioniert, keine Struktur
  • 🔒 Cursor Rules — Cursor-only, kein Gedächtnis, keine Fehler-Lernkurve
  • 🧊 mem0 / MemGPT — Opaque (du siehst nicht was gespeichert wird), API-abhängig

→ Keiner dieser Ansätze LERNT aus Fehlern.

Die Lösung: Skill-Everythink

  • ✅ Lebendes Skill-System aus Plain Markdown — kein Setup, kein API-Key
  • ✅ Fehler-Gedächtnis — Jeder Fehler wird festgehalten, analysiert, zur Regel
  • ✅ Domänenwissen — Prozesse, Regeln, interne Begriffe, Partner, Architektur
  • ✅ Git-versioniert — Jede Änderung nachvollziehbar, reviewbar, rollback-fähig
  • ✅ Agent-agnostisch — OpenCode, Claude, Cursor, jeder Agent der Kontext liest
  • ✅ Modular — Sub-Skills für verschiedene Domänen, wächst mit dem Team

→ Wie Fine-Tuning, aber kostenlos, flexibel und transparent.

Architektur

skill-everythink/ ├── SKILL.md ← Router / Einstiegspunkt ├── references/ │ ├── development/ ← Code-Qualität, 15 Regeln │ │ └── code-quality.md │ ├── git/ ← Commit-Konventionen │ │ └── conventions.md │ ├── domain/ ← Firmenwissen (Template) │ │ └── template.md │ ├── process/ ← Reviews, Deployment │ │ └── review-deployment.md │ └── errors/ ← Fehler-Log + Workflow │ ├── error-log.md │ └── self-extension-workflow.md └── _templates/ ← Vorlagen für neue Skills

SKILL.md agiert als Router — der Agent lädt nur die Sub-Skills, die er gerade braucht.

Selbst-Erweiterung

Der Agent erweitert den Skill automatisch:

1
Fehler erkennen — Test schlägt fehl, Review-Kommentar, falscher Ansatz
2
Kategorie bestimmen — Development, Git, Deployment, Security, Domain?
3
"Search before Write" — Gibt es schon eine ähnliche Regel? Update statt Duplikat.
4
Regel formulieren — Handlungsanweisung, nicht Beschreibung
5
In Ziel-Datei einfügen — Am Ende des passenden Abschnitts
6
Error-Log aktualisieren — Vollständiger YAML-Eintrag mit Kontext
